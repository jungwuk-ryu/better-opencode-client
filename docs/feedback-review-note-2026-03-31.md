# 피드백 리뷰 노트

작성일: 2026-03-31  
목적: 공개 후 들어오는 피드백을 즉시 backlog에 잘 넣기 위한 기준을 남긴다.

## 1. 리뷰 원칙

- 모든 요청을 즉시 구현 약속으로 바꾸지 않는다.
- 먼저 사용자 영향과 재현성을 본다.
- 비슷한 피드백은 묶어서 본다.

## 2. 분류

| 분류 | 의미 | 다음 행동 |
| --- | --- | --- |
| 버그 | 현재 동작이 깨짐 | 재현 후 수정 우선순위 지정 |
| 사용성 | 기능은 되지만 어렵다 | 문구, 배치, flow 조정 |
| 누락 | 기대한 경로가 없다 | 범위 검토 후 roadmap 반영 |
| 탐색 | 있으면 좋겠지만 지금은 실험 | experimental로 분리 |

## 3. 리뷰 질문

1. 이 피드백은 몇 명이 같은 방식으로 겪을 가능성이 있는가?
2. 현재 핵심 루프를 막는가?
3. 문서나 교육으로 먼저 해결할 수 있는가?
4. 제품 방향을 바꾸는 수준인가, 아니면 표현 문제인가?

## 4. backlog 반영 방식

- P0는 연결, 완료 루프, 데이터 안정성
- P1은 주요 운영 흐름
- P2는 문서와 UI 정리
- experimental은 별도 라벨로 묶는다

## 5. 회수 후 정리

- 어떤 피드백이 문서 수정으로 해결됐는지 적는다.
- 어떤 피드백이 코드 수정으로 해결됐는지 적는다.
- 어떤 피드백이 범위 밖이어서 보류됐는지 적는다.
