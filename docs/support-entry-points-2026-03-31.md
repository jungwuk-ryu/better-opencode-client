# 지원 진입점

작성일: 2026-03-31  
목적: 사용자가 막혔을 때 어디로 가야 하는지 단순하게 정의한다.

## 1. 지원 원칙

- 사용자는 먼저 문제를 재현할 수 있어야 한다.
- 지원 경로는 가볍고 찾기 쉬워야 한다.
- 새 기능보다 먼저 현재 실패 상황을 기록한다.

## 2. 현재 지원 경로

| 경로 | 언제 쓰는가 | 기대 결과 |
| --- | --- | --- |
| GitHub issue | 버그 재현, 기능 요청, 문서 오류 | 추적 가능한 backlog item |
| Pull request comment | 변경 중인 기능에 대한 빠른 피드백 | 구현 컨텍스트 안에서 논의 |
| 릴리즈 노트 피드백 | 릴리즈 후 사용자 체감 문제 | 다음 패치 우선순위 입력 |
| 수동 검증 체크리스트 | QA 또는 재현 확인 | 원인 분리와 재현성 확보 |

## 3. issue 작성 가이드

issue는 아래 5개를 기본으로 적는다.

1. 어떤 화면에서 발생했는가
2. 무엇을 하려 했는가
3. 실제로 무엇이 일어났는가
4. 예상한 결과는 무엇이었는가
5. 로그, 스크린샷, deep link, 계정/서버 정보가 있는가

## 4. 우선순위

- `P0`: 연결 불가, 데이터 손실 가능성, 핵심 루프 차단
- `P1`: 주요 작업 흐름 지연
- `P2`: 문구, 레이아웃, 품질 개선

## 5. 운영 메모

- 지원 경로는 너무 많이 늘리지 않는다.
- 처음에는 GitHub issue 한 곳으로 시작해도 충분하다.
- 중요한 것은 경로의 수가 아니라 재현 정보의 질이다.
